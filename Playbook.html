Production Deployment Checklist

#deployment #release #weekly #engineering

Run this before every production push. The big idea: nothing surprises you in prod. We expect a green build, a clean canary, and a rollback we’ve actually rehearsed. If any item below isn’t green, the release waits.

Do
7
Deploy to canary first; soak for 20 minutes minimum.
Watch the four golden signals dashboard during rollout.
Post a #releases note with the diff link and on-call name.
Run the smoke‑test playbook end‑to‑end after deploy.
Page yourself on critical alerts for the first hour.
Capture and pin the rollback command before merging.
Update the changelog and bump the user-facing version.
Don’t
5
Don’t deploy on Fridays after 2pm. Past Mara has feelings about this.
Don’t skip the canary, even for “tiny” config changes.
Don’t bundle a schema migration with a feature ship.
Don’t silence alerts to “reduce noise” mid‑release.
Don’t merge to main without a paired reviewer awake.
Notes from the last release

The new index made the canary CPU spike for ~3 minutes before settling. That’s expected and not a regression — leave it alone, but note it in the runbook so the next on‑call doesn’t panic. Also: the #releases bot is back online, no need to manually post anymore.
